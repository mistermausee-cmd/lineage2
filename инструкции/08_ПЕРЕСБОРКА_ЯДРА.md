## 2. БИЛД «под ключ» (как пересобрать GameServer.jar)

```bash

cd mobius-src
git sparse-checkout set L2J_Mobius_04.0_GrandCrusade
# Фиксируемся на нашей ревизии:
git checkout 88b6c5f4221eb64755e0ce3f3e3f55feec171efd
```

### 2.2. Инструменты (уже стоят в рабочей среде, в `/projects/sandbox/_tools/`)
- **JDK 25**: `/root/.local/share/mise/installs/java/25`
- **Apache Ant 1.10.15**: `/projects/sandbox/_tools/apache-ant-1.10.15`

### 2.3. КОМАНДА СБОРКИ (собирает только jar-ы, быстро)
```bash
cd /projects/sandbox/mobius-src/L2J_Mobius_04.0_GrandCrusade
JAVA_HOME=/root/.local/share/mise/installs/java/25 \
PATH=$JAVA_HOME/bin:/projects/sandbox/_tools/apache-ant-1.10.15/bin:$PATH \

#        (полный dist zip: ../build/*.zip — цель `ant`, но для подмены jar он не нужен)
```

> `ant jar` — быстрая цель (только компиляция классов в jar).
> `ant` (без цели) — полный дистрибутив zip (дольше, нужен только для чистой установки с нуля).

### 2.4. Подмена jar в репозитории
```bash
cp /projects/sandbox/mobius-src/L2J_Mobius_04.0_GrandCrusade/../build/dist/libs/GameServer.jar \
   /projects/sandbox/lineage2/server/libs/GameServer.jar
# затем commit + push в main (см. §6)
```
На сервере игрок просто делает `git pull --ff-only origin main` и `bash start_linux.sh` — новый jar подхватывается автоматически (jar не в .gitignore).

## 3. ХИРУРГИЧЕСКИЕ ПРАВКИ ЯДРА (что именно изменено vs upstream)

От оригинальной ревизии `88b6c5f4` наш jar отличается **правками в 2 классах**. Всё остальное — стоковый Mobius.

### 3.1. Файл `java/org/l2jmobius/gameserver/model/item/instance/Item.java`

**(A) Новый метод `reapplyActiveSkillsToOwner()`** — ФИКС БАГА КОЛЕЦ БОГОВ.

Симптом (репорт игрока): при перезаходе в игру умения колец богов (и прочей эпик-бижи с
активными/ON_EQUIP скиллами) становятся НЕАКТИВНЫМИ, будто предмет не надет. Снять/надеть кольцо —
умение снова доступно.

Причина: на логине paperdoll-listener пропускает предметы с ограничением по грейду (expertise level
ещё не выставлен во время restore инвентаря), а `giveSkillsToOwner()` восстанавливает только
ПАССИВНЫЕ скиллы. Активные скиллы предметов (тоглы бижи богов) остаются не выданными до ручного
переодевания.

Решение: отдельный метод, вызываемый **только на логине** (см. 3.2), с защитой от дублей.
НЕ трогали `giveSkillsToOwner()`, чтобы не вызвать двойной ON_EQUIP при обычной экипировке.

```java
/**
 * Re-applies this equipped item's ACTIVE skills (toggles / usable abilities) and ON_EQUIP self-buffs to the owner.
 * On login the paperdoll listener skips grade-restricted items ... only PASSIVE skills are re-added, so active
 * item skills (e.g. god's jewelry toggles) would otherwise be missing until the item is manually re-equipped.
 * This method is meant to be called on player login only (see PlayerInventory.applyItemSkills).
 */
public void reapplyActiveSkillsToOwner()
{
    if (!isEquipped())
    {
        return;
    }
    final Player player = asPlayer();
    if (player == null)
    {
        return;
    }
    final boolean[] updated = { false };
    _itemTemplate.forEachSkill(ItemSkillType.NORMAL, holder ->
    {
        final Skill skill = holder.getSkill();
        if ((skill != null) && !skill.isPassive() && (player.getSkillLevel(skill.getId()) < skill.getLevel()))
        {
            player.addSkill(skill, false);
            updated[0] = true;
        }
    });
    _itemTemplate.forEachSkill(ItemSkillType.ON_EQUIP, holder ->
    {
        final Skill skill = holder.getSkill();
        if ((skill != null) && !player.isAffectedBySkill(skill.getId()))
        {
            skill.activateSkill(player, player, this);
        }
    });
    if (updated[0])
    {
        player.sendSkillList();
    }
}
```

**(B) Новый метод `setRemainingTime(long milliseconds)`** — для выдачи временных копий постоянных предметов (руны/бустеры на 3 дня от NPC Теодора, см. `01_ЧТО_СДЕЛАНО`).

Позволяет выдать предмет с ОРИГИНАЛЬНЫМ item-id, но ограниченным сроком жизни (3 дня),
не создавая кастомных id и клиентских записей. Значение пишется в БД, задача истечения перепланируется.

```java
/**
 * Overrides the remaining lifetime of this item and reschedules its expiration.
 * Allows granting a time-limited copy of an item whose template is permanent, e.g. 3-day booster runes from an NPC.
 * @param milliseconds new remaining lifetime; a value <= 0 makes the item permanent
 */
public void setRemainingTime(long milliseconds)
{
    _time = (milliseconds <= 0) ? -1 : (System.currentTimeMillis() + milliseconds);
    _storedInDb = false;
    updateDatabase(true);
    ItemLifeTimeTaskManager.getInstance().remove(this);
    scheduleLifeTimeTask();
}
```

### 3.2. Файл `java/org/l2jmobius/gameserver/model/itemcontainer/PlayerInventory.java`

В методе `applyItemSkills()` (вызывается на логине) добавлен вызов нового метода для надетых предметов:

```java
public void applyItemSkills()
{
    for (Item item : _items)
    {
        item.giveSkillsToOwner();
        item.applyEnchantStats();
        if (item.isEquipped())
        {
            item.applySpecialAbilities();
            item.reapplyActiveSkillsToOwner(); // Restore active/ON_EQUIP item skills (god's jewelry toggles) lost on relogin.
        }
    }
}
```

> Баг колец = **серверный** (логика восстановления скиллов на логине), не клиентский. Проверена вся
> бижа богов разных уровней — корень один и тот же, фикс общий для всех активных item-скиллов.

### 3.3. Отдельная категория множителей для ГРАНД-БОССОВ (эпиков)

Задача: ослабить эпиков (Antharas ~689М HP, ~3М P.Atk и т.д.) сильнее обычных РБ под соло.

Проблема в стоке: в `NpcTemplate.java` типы `RaidBoss` и `GrandBoss` шли в ОДНОМ `case` и получали
одни множители `RAIDBOSS_*`. Отделить эпиков без правки ядра было нельзя.

**Правка (2 файла):**

`java/org/l2jmobius/gameserver/config/custom/NpcStatMultipliersConfig.java` — добавлены 8 полей
`GRANDBOSS_*` и их загрузка. Fallback: если ключ `GrandBoss*` отсутствует в ini, берётся значение
соответствующего `Raidboss*` (обратная совместимость):
```java
GRANDBOSS_HP_MULTIPLIER = config.getDouble("GrandBossHP", RAIDBOSS_HP_MULTIPLIER);
GRANDBOSS_PATK_MULTIPLIER = config.getDouble("GrandBossPAtk", RAIDBOSS_PATK_MULTIPLIER);
// ... MP / MAtk / PDef / MDef / AggroRange / ClanHelpRange аналогично
```

`java/org/l2jmobius/gameserver/model/actor/templates/NpcTemplate.java` — `case "GrandBoss"` отделён
от `case "RaidBoss"` и использует `GRANDBOSS_*` множители (HP/MP/PAtk/MAtk/PDef/MDef/aggro/clanHelp).

**Значения (в `server/game/config/Custom/NpcStatMultipliers.ini`):** `GrandBossHP=0.15`,
`GrandBossPAtk/MAtk=0.30`, `GrandBossPDef/MDef=0.50`. ⚠️ Сами числа меняются в ini БЕЗ пересборки
(только рестарт) — пересборка нужна была лишь чтобы завести саму категорию.

Проверка типов: эпики (`type="GrandBoss"`): Antharas 29068, Valakas 29028, Queen Ant 29001, Core 29006,
Orfen 29014, Baium 29020, Beleth 29118, Frintezza 29045, Lindvior 29240 и др. ВНИМАНИЕ: часть «гранд»-боссов
из телепорт-списка на деле `type="RaidBoss"` (напр. Sailren 29065) — на них действует `Raidboss*`, не `GrandBoss*`.

### 3.4. Отвергнутые варианты (чтобы не повторять)
- Фикс в `giveSkillsToOwner()` — отвергнут: вызвал бы двойной ON_EQUIP при обычной экипировке.
- Декомпиляция + правка `Item.java` из jar через CFR — отвергнута: декомпилят не компилировался.
- Javassist / рефлексия в рантайме — отвергнуто: непроверяемо и хакерски.
- 3-дня рун через кастомные item-id — отвергнуто: нужны клиентские записи (dat). Вместо этого —
  core-метод `setRemainingTime` на оригинальных id.

## 4. ЧТО ВЛЕЧЁТ ПЕРЕСБОРКА ЯДРА (ответы на вопросы игрока)

- **Все прежние правки конфигов/данных/скриптов/магазина СОХРАНЯЮТСЯ.** Пересборка меняет только
  `GameServer.jar` (Java-логику). Конфиги (`.ini`), данные (`data/**`), HTML Alt+B, мультиселлы,
  скрипты — это отдельные файлы, jar их не перезаписывает.
- **Обновление «по-тихому»:** игрок просто делает `git pull --ff-only origin main` + рестарт.
  Новый jar подхватывается, всё остальное на местах, баги пофикшены.
- **Предметы игрока НЕ пропадают** от пересборки (они в БД). 3-дневные руны Теодора истекают по своему
  сроку (`setRemainingTime`); предметы из магазина — постоянные, НЕ истекают.
- **Фикс плаща и прочие правки данных** сохраняются (они не в jar).

- Проверять «описание ↔ фактический эффект» и упираться в КАПЫ (см. ниже), прежде чем звать баг багом.

- `MaxPSkillCritRate = 2000`

### 5.2. Разобранные «не-баги»
- Диадема Мудрости «+50 к шансу маг.крита», а в статах 200 — это КАП `MaxMCritRate=200`, не баг.
